# Agent Orchestration (エージェント・オーケストレーション)

**Activation:** This rule is **ALWAYS ON** for all files (`**/*`).

メインセッション（Fable 5 / Opus）と実装サブエージェントの役割分担を定義する。
目的は **トークンの節約** と **設計・実装の責務分離** であり、メインセッションの
コンテキストを設計・監査・レビューに集中させることにある。

## 1. メインセッションの責務 (Architect / Auditor)

- メインセッション（Fable 5 または Opus）は **設計・監査・レビュー・計画** に専念する。
- 自らの長いコンテキストを実装作業の試行錯誤で消費しない。
- 具体的には次を担う:
  - 要件の分解と実装方針の設計。
  - サブエージェントへ渡す **仕様（対象ファイル・変更内容・検証方法）** の作成。
  - サブエージェントの成果（差分）の **監査（レビュー）** とコミット判断。

## 2. 実装の委譲ポリシー (2階層 + 例外)

実装は原則としてサブエージェントへ委譲し、難易度で使い分ける。

| 難易度 | 委譲先 | 使用モデル |
|---|---|---|
| ルーチン・標準実装 | `implementer-sonnet` サブエージェント | Sonnet |
| 難易度が高い実装 | `implementer-opus` サブエージェント | Opus |
| 最難所（高度な設計判断を伴う） | **メインセッションが直接実装してよい** | Fable 5 / Opus |

- 委譲は **Task ツール（サブエージェント起動）** で行い、コンテキストを分離してトークンを節約する。
- 迷ったら「まず Sonnet、失敗・難航したら Opus、それでも難しければメイン直接」とエスカレーションする。

## 3. 委譲時の渡し方 (Hand-off Contract)

- サブエージェントには **完結した仕様** を渡す。最低限、次を明示する:
  - 対象ファイル（パス）。
  - 具体的な変更内容（擬似コード・シグネチャ・期待挙動）。
  - 検証方法（実行すべきテスト・コマンド）。
- サブエージェントに **設計判断（アーキテクチャ選定・仕様決定）をさせない**。判断が必要になったら
  メインに差し戻させる。
- サブエージェントは作業完了後、**変更点・検証結果・未対応事項を簡潔に報告** する（差分全文の貼り付けは不要）。

## 4. 監査ゲート (Audit Before Commit)

- サブエージェントの成果は、メインが **必ず差分を監査** してからコミットする。
- 無監査でのコミット・プッシュは禁止。サブエージェントの結果を盲信しない。

## 5. 既存ルールとの整合

- 委譲はあくまで **改修（Do）フェーズ** の手段であり、`plan-before-modify` を上書きしない。
  実装着手は **ユーザー承認後** に限る。
- コミットは `git-commit-rules.md`、記録は `plan-before-modify.md`（`docs/history/` + `CHANGELOG.md`）に従う。
- このオーケストレーションを一連のサイクルとして回す手順は、PDCA ワークフロー
  （`.claude/commands/pdca.md`）に定義する。
